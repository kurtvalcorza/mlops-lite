# Research: 020 Stack Remediation

Decisions R1–R8. Every externally sourced claim about the candidate store is re-verified by the
US1 **spike** before migration — the spike, not this document, is the gate (FR-202).

## R1. Replacement store: Garage first, SeaweedFS fallback

- **Decision**: Default candidate **Garage** (pinned image digest, single node,
  `replication_factor = 1`). Fallback **SeaweedFS** if the spike finds a compatibility gap the
  configuration can't close. The decision executes via the spike, not by fiat: pass ⇒ Garage;
  fail ⇒ repeat the identical spike/migration/cutover plan against SeaweedFS.
- **Rationale**: MinIO's OSS edition is archived upstream (read-only, no patches) — the exit
  itself is settled. Garage is the Principle-III fit: a single small static binary designed for
  exactly this "one boring S3 daemon on one box" role, with idle RSS in the tens of MB (spike
  measures; SC-130 gates). Its AGPL license matches the incumbent's and is acceptable for this
  internal, non-redistributed platform (spec Assumptions). SeaweedFS is more featureful, Apache-2.0,
  and the ecosystem's drop-in favorite — but it is a larger system (filer + volume architecture)
  than one machine needs; it earns the fallback slot, not the default.
- **Alternatives considered**: RustFS (alpha — explicitly not production-ready; rejected);
  Ceph/RGW (multi-node machinery, absurd weight here); plain filesystem + no S3 API (would orphan
  MLflow artifact storage and every boto3 seam — a rewrite, not a swap); staying on the frozen
  MinIO digest (indefinitely unpatched storage under the whole platform — the finding 020 exists
  to retire).

## R2. Garage single-node bootstrap (`infra/garage/`)

- **Decision**: `infra/garage/garage.toml` (metadata + data dirs on the compose volume,
  `replication_factor = 1`, `s3_region` set to the SAME value the clients already send
  (`AWS_DEFAULT_REGION=us-east-1`) so no client-side region change is needed) plus a one-shot
  `infra/garage/init.sh` compose service (the `createbuckets` replacement) that runs the garage
  CLI inside the container: node-ID discovery → `layout assign` (zone `local`, capacity from the
  drive budget) → `layout apply` → `bucket create` ×4 (`datasets`, `models`, `results`,
  `mlflow`) → `key create` → per-bucket `allow --read --write` — all idempotent (re-runs no-op).
  Credential flow runs the REVERSE direction vs MinIO (contracts/store-migration.md §bootstrap):
  the garage CLI mints the key pair at bootstrap and `init.sh` emits it; `scripts/gen_secrets`
  records it into `.env` (MinIO's pair was generated by gen_secrets and fed INTO the store).
- **Rationale**: mirrors the existing `infra/{mlflow,prometheus}` + one-shot-bootstrap house
  pattern; keeping the client-visible region/addressing identical confines the swap to
  `MLFLOW_S3_ENDPOINT_URL` + credentials. Path-style addressing (what every client already uses
  against a custom endpoint) is Garage's default S3 mode; the spike pins it.
- **Spike checklist (FR-202, runs before any migration)**: MLflow artifact round-trip incl. a
  multi-hundred-MB model file (multipart upload path); `platformlib.store` list_keys /
  list_common_prefixes pagination past 1,000 keys; delete-idempotency and the 404 discrimination
  `_missing()` relies on; write-once duplicate-PUT behavior `attach_label` relies on; presigned
  URLs explicitly NOT required (serve-artifacts proxy — spec assumption re-verified here); idle
  RSS measurement at rest (SC-130); offline suite + golden flows green against the candidate —
  run via a TEMPORARY env-seam flip to the empty candidate (the flows self-create their data),
  flipped back before migration; the mini-flip doubles as a rehearsal of the cutover contract.

## R3. Migration tool: our own boto3 mirror, not rclone/mc

- **Decision**: `scripts/migrate_store.py` — boto3 (already a pinned dependency), source→dest
  streaming copy by bucket/prefix, **idempotent** (skip when key exists with equal size),
  **bidirectional** (`--reverse` for the rollback re-mirror), emitting the JSON **migration
  report** (per-bucket object counts + byte totals on both sides, direction, timestamp) that
  gates cutover (FR-199/SC-127). Parity compares **count + bytes**, deliberately not ETags —
  multipart ETags are not portable across S3 implementations.
- **Rationale**: zero new tools; unit-testable offline with two FakeS3 instances (house seam
  style); `mc` is MinIO's own now-archived tooling (using the dying tool to escape the dying
  store is self-defeating); rclone is fine software but an unpinned new dependency for a one-shot
  job our existing client library does in ~150 lines with a report shaped exactly how the spec
  wants it.
- **Alternatives considered**: `mc mirror` (archived upstream, rejected); `rclone sync`
  (maintained but new tooling + report post-processing; rejected as unnecessary); MLflow
  artifact-level export/import (covers one bucket of four; rejected).

## R4. Cutover mechanics: one env seam, both stores live through the window

- **Decision**: The store endpoint is an env seam (the AWS credential pair plus the endpoint,
  which resolves as `S3_ENDPOINT_URL` → `MLFLOW_S3_ENDPOINT_URL` → a hardcoded `minio:9000`
  fallback) consumed by the gateway container, the MLflow server, and the host venv.
  Sequence: garage service up alongside minio → spike → migrate → **flip the env + restart**
  (cutover) → soak (golden flows + normal operation) → operator confirms → delta re-mirror
  back-check → remove minio + createbuckets + pinned digests + secrets wiring + docs + the
  hardcoded source-default endpoints (FR-201). Rollback at any pre-confirm point: flip the env back
  (FR-200); a `--reverse` mirror carries any post-cutover writes back first.
- **Seam caveat (the review's finding — the seam is two vars + baked defaults, not one)**: because
  `S3_ENDPOINT_URL` wins over `MLFLOW_S3_ENDPOINT_URL`, and the host consumers bake a `:9000`
  default (`hostagent/run.sh`, `training/flows/*`), a naive "flip one var" cutover can silently
  no-op (if `S3_ENDPOINT_URL` is set) or break on the wrong port (if a host consumer falls back to
  its default). So the cutover asserts `S3_ENDPOINT_URL` unset, exports the endpoint explicitly to
  the Garage host port, and verifies the client's *resolved* `endpoint_url` moved
  (contracts/store-migration.md §cutover). The clean single-seam story remains the design intent;
  020 hardens the two rough edges rather than assuming them away.
- **Rationale**: the platform earned this cheap cutover by routing every byte through one
  endpoint seam since 003 — 020 collects on that design. Compose keeps both services defined
  during the window; no profiles machinery needed (the seam is the switch, the services are just
  present), which also keeps `docker compose config` unconditional.

## R5. Children framework: FastAPI + uvicorn, one pin shared with US3

- **Decision**: `serving/children/{vision,embed,tabular}_service.py` — FastAPI apps with exactly
  the routes the Bento children serve today (same paths, same payloads, same `/readyz`, same
  dynamic-port launch contract), run by uvicorn from per-child `run.sh` unchanged in shape. One
  `fastapi`+`uvicorn` pin enters the host venv, shared by US3's optional agent runtime.
- **Rationale**: multipart image upload (vision) is painful on the stdlib (`cgi` is gone in
  3.13); FastAPI is already the platform's HTTP idiom (gateway) and `bentoml` transitively ships
  starlette+uvicorn anyway — so the swap strictly shrinks the dependency tree while adding zero
  unfamiliar technology. Model code (torch/torchvision MobileNet, sentence-transformers,
  LightGBM) moves over verbatim.
- **Alternatives considered**: stdlib `http.server` children (zero deps but hand-rolled
  multipart parsing — rejected for the same reason the agent doesn't hand-parse multipart
  bodies it merely relays); keeping Bento only for vision (splits the venv story, keeps the
  framework for one route — rejected).

## R6. Golden-set gate for the child swaps

- **Decision**: `scripts/capture_goldens.py` records, per child, the request/response pairs for
  the canonical verbs (vision classify with a fixture image, embed, tabular predict) plus the
  readiness probe — status, headers-that-matter (content type), and body bytes — into
  `tests/goldens/<engine>/`. The swap gate replays them against the new child and diffs
  byte-for-byte at the agent boundary (FR-203/SC-131). Capture runs against the live stack
  (goldens are environment artifacts, not committed model outputs — floats can be
  hardware-sensitive); offline, the existing adapter/contract tests keep covering the agent side
  with fake children.
- **Rationale**: "byte-compatible" is only enforceable with recorded bytes; the agent boundary is
  the right observation point because it is the only consumer (FR-177 made children opaque).

## R7. Agent runtime: dual-runtime switch + drill, uvicorn pre-approved

- **Decision**: `AGENT_RUNTIME=stdlib|uvicorn` (default `stdlib` until the verdict). The ASGI app
  reuses the existing framework-free forward functions (`forward_engine`,
  `forward_engine_multipart`, jobs/health/metrics handlers) — the handler logic does not fork;
  only the transport does. `scripts/agent_stream_drill.py` measures on hardware: stream
  time-to-first-token and inter-token stall count under concurrent health polling; multipart
  round-trip latency; mid-stream client-disconnect recovery; behavior with a swap/preempt arriving
  during a stream. Baselines: the 017/018 runbook numbers. Verdict recorded in
  `docs/on-hardware-validation.md` (the same doc the pending [HW] sweep writes). Miss ⇒ flip
  the default to `uvicorn`, re-run drill + full agent suite; meet ⇒ stdlib stays and the switch is
  removed at the following increment (no dead config left behind).
- **Rationale**: the spec demands evidence, not fashion (FR-205); the dual switch is what makes
  the drill an A/B on the same box in one session. Reusing the forward functions keeps Principle
  II analysis unchanged — the six-times-reviewed admission/lifecycle code is not re-entered.
- **Alternatives considered**: unconditional uvicorn migration (churns a heavily-reviewed surface
  with no evidence it's needed — rejected by FR-205); aiohttp (second HTTP idiom in the codebase —
  rejected); keeping both runtimes forever (dead config + doubled test matrix — rejected; the
  loser is deleted).

## R8. GPU-budget portability audit + bring-up doc

- **Decision**: A repo audit pins that the only VRAM-budget literals are `VRAM_GB` env fallbacks.
  Today that default (`"12"`) is duplicated across five reads (`hostagent/main.py`,
  `hostagent/jobs.py`, and the `llama`/`vision`/`whisper` adapters); the audit **consolidates them
  to one resolver** so a machine set to a different budget can't be left partially on the old
  value if the env is ever unset. Model-size estimates and per-engine idle knobs are already
  env-driven; live NVML reads remain the primary admission input (static budget = fallback only). The bring-up
  checklist (new-machine section in the README refresh that T379 already owes) lists: `VRAM_GB`,
  native binary rebuilds (llama.cpp / whisper.cpp `build.sh`), the CUDA-indexed torch/torchvision
  wheels, `scripts/gen_secrets`, and the state-dir/beacon behavior on a renamed host (self-heals
  when idle — 018).
- **Rationale**: cheap insurance, mostly true by construction since 018; SC-133 makes it checked
  rather than assumed, and co-locating the doc with T379's refresh avoids two competing README
  edits.
